# Reference Analysis

Use this reference when gathering moodboards, competitor screens, app UI examples, game UI screenshots, or design-system examples.

## Source Quality

- Prefer official design systems, documented product patterns, accessibility standards, research libraries, and shipped products over purely aesthetic galleries.
- Treat Dribbble, Behance, Pinterest, and social posts as visual inspiration only. They rarely prove usability, state coverage, accessibility, or production constraints.
- Keep app UI and game UI references in separate pools because density, input model, attention, and success criteria differ.
- Do not copy a reference. Extract reusable decisions: layout, spacing scale, color roles, state logic, hierarchy, input model, and motion purpose.

## Tags For Every Reference

Record:

- Product type: app, dashboard, mobile, web, game, tool, editor, or store.
- Surface: home, list/detail, form, table, settings, HUD, inventory, shop, pause menu, dialog, onboarding, or empty/error state.
- User type: first-time, returning, expert, casual player, competitive player, admin, operator, or customer.
- Task type: browse, decide, compare, configure, recover, purchase, play, monitor, or act under time pressure.
- Input type: touch, mouse, keyboard, controller, gamepad, remote, or mixed.
- What to transfer: hierarchy, grouping, state model, density, typography, color role, icon style, or motion logic.
- What not to copy: brand, logo, character art, exact composition, proprietary assets, or source-specific style.

## App UI Deconstruction

- Analyze navigation level, scan path, data density, primary action placement, filtering/search behavior, and state coverage.
- Check whether the reference supports repeated use or only looks impressive as a static screen.
- Evaluate forms by labels, examples, validation, error recovery, and optional-field handling.
- Evaluate dashboards by decision usefulness, not chart decoration.

## Game UI Deconstruction

- Analyze whether the UI is diegetic, non-diegetic, spatial/geometric, or meta.
- Check how much of the gameplay camera is blocked.
- Identify which information is glanceable and which is pushed to pause/menu/inventory.
- Check touch/controller focus, selected state, button prompts, remapping, subtitles, colorblind support, and readable HUD contrast.

## Validation

- Every chosen reference must have a one-sentence "why this helps the user's task" note.
- Every reference analysis should mention at least one state: empty, loading, error, focus, selected, disabled, or pressed.
- If a reference is copyrighted or license-unknown, keep it analysis-only and do not embed it into the skill package.

## Sources

- Baymard Institute, UX research library and ecommerce UX examples.
- Apple Human Interface Guidelines.
- Google Material Design / Material 3.
- Microsoft Fluent.
- W3C, WCAG 2.2.
- Optional: any research the user supplies for their own project (treat as local grounding, not a bundled source).
